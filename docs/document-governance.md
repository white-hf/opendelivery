# 双产品文档与开发评审流程

## 分类规则

- `docs/driver/`：司机 App/Driver API 专属产品、设计、迭代、测试、运行和总结。
- `docs/operations/`：Operations Web/API 专属文档。
- `docs/shared/`：两个产品共同依赖的状态机、数据模型、集成和联合 Gate 索引。
- 原 `docs/prd|design|iterations|summaries` 是历史和共享资料库；权威入口由三个产品中心标明。现阶段不批量移动旧文件，避免破坏链接和历史追溯。

## 强制交付顺序

1. **落地文档**：先更新对应产品 PRD；涉及契约、状态、表或跨产品事件时同步更新设计/API/数据模型。
2. **迭代评审**：迭代文档写明范围、非目标、依赖、迁移、兼容、风险、测试和 DoD，状态从 `DRAFT` 变为 `REVIEWED`。
3. **开发**：只实现已评审范围。发现需求变化先回到文档，不在代码中隐式扩展产品边界。
4. **测试**：单元测试 → 真实 MySQL 集成 → API/客户端契约 → 浏览器或模拟 App → 跨产品联合 E2E。
5. **总结与发布**：Summary 记录真实交付、命令、指标、遗留风险和回滚；满足 Gate 后才标记 `COMPLETE`。

## Review 清单

- 产品归属和另一产品的只读/命令边界是否明确？
- API 是否给出完整输入、输出、错误、权限、幂等和版本策略？
- 状态与 custody 是否守恒，是否存在客户端直接指定身份或目标状态？
- 数据迁移是否只新增 Flyway 文件，是否有索引、容量和历史数据策略？
- 是否覆盖三城市隔离、三语言、401/403/409、重复、并发和失败恢复？
- 联合流程是否说明由 Driver 还是 Operations 发命令，以及对方如何观察/批准？

小型缺陷可用精简评审记录，但不能跳过“文档 → Review → 开发 → 测试”。紧急修复应在同一提交补齐决策和回归证据。
