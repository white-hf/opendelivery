# R02.1 运营控制塔迭代计划

## 目标与优先级

这是三城市试运营前的必做版本，解决“运营不知道当前阶段、下一步动作和地图为何为空”。不加入路线优化等非必要能力。

| 小迭代 | 可验收结果 | 主要任务 | 完成证据 |
|---|---|---|---|
| CT01 流程导航 | 菜单按营业日顺序工作 | 导航分组、阶段状态/徽标、当前步骤、配置分离、深链接筛选 | 三角色任务寻路；三语言 |
| CT02 今日控制塔 | 首页可判断和行动 | 统一指标、流程条、运力、异常、下一步队列、数字钻取 | 聚合=明细；三站隔离 API/UI |
| CT03 地图可观察性 | 有坐标必见点 | 自动 fit、聚合、查询/可定位/显示计数、图例筛选、明确空态/错误态 | Google 浏览器验收；69/72 fixture 口径 |
| CT04 业务旅程回归 | 可沿导航完成一天 | 跨工作台上下文、保持视口/筛选、Playwright、可访问性 | YHZ/YYZ/YVR E2E 与运营签字 |

## 技术方法

- Control Tower API 返回稳定 `code/count/severity/target/filters`，前端不重新拼装业务口径。
- 阶段状态从事实计算，不增加人工“当前阶段”字段。
- URL 保存 `station/serviceDate/view/filter`；切站取消旧请求并清 Query Cache。
- 地图 API 返回查询总数、可定位数、异常数和点列表；生产按视口/聚合级别查询。
- 测试订单使用正式 Waybill/Parcel/Geocode/Area Assignment 表，禁止前端硬编码点位。

## Definition of Done

四个切片顺序交付并独立演示。API、产品、设计、测试、总结中英文同步。API 有数据不代表地图完成；必须在浏览器看到点位、计数一致并验证三站切换。
