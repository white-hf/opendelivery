# O05 交接审批迭代

> 状态：`COMPLETED`（2026-07-21 完成）；归属：运营产品；对应双产品计划 O05，是联合发布 Gate 的核心卡点。已关联 [执行总结](../summaries/iteration-o05-handover-approval-summary.md)。

## 可运营结果

主管对司机提交的装车扫描做全批次核对与批准：只有正确司机的有效扫描覆盖的包裹才能把 custody 从站点转给司机；差异有决定、拒绝可重开、并发批准只有一个成功。现有 `/ops/v1/scan-sessions/{id}/approve` 雏形在本迭代加固为完整门禁。

## 范围

1. **全批次门禁**：同一波次下全部必要 session 已提交（或主管例外关闭）后，才允许批准其中任一 session；无有效扫描（valid=0）的 session 禁止批准；门禁不通过给出可执行提示（缺哪些 session/司机）。
2. **差异决定**：对漏扫、错扫、未提交逐件或按类别决定——放行（例外关闭）、退回重扫（session 由 SUBMITTED 置回 OPEN，司机本人补扫）、登记线下交接备注；全部决定留审计。
3. **原子 custody 转移**：批准在单事务内完成——session 置 APPROVED、task item 置 LOADED、parcel custody 转 DRIVER 并追加 `custody_event`/`parcel_status_event`/`operation_audit_log`/`outbox_event`；任一步失败全回滚，不产生半截状态。
4. **并发唯一成功**：session 行悲观锁 + 前置状态校验（仅 SUBMITTED 可批准）；并发批准/重复提交幂等返回首次结果或 409，绝无双重 custody 转移。
5. **安全改派**：批准前发现的错任务包裹可改派到正确司机的本日任务（复用规划 `reassign` 的事务与校验），改派留 custody 语义痕迹。
6. **前端**：交接审批工作台（按波次列 session、门禁状态、差异清单、批准/退回/改派动作、审批结果），三语言；导航"交接审批"进入。

## 非目标

- 不改司机扫描与提交（D02）；不做 RETURN/退站审批（O06）；不做电子签字（O07 的签字概念独立）。
- 不做部分批准：一次批准作用于整个 session 的有效集合，差异只能决定不能拆开批准。

## 依赖

- V1 扫描/custody/outbox 主链、V6 交接门禁、规划波次任务（R04 之后以其产出的任务/应扫清单为准）、O04 监督口径（同源数字）。

## 迁移与兼容

- 预计无新表；如需 session 决定明细表（差异决定逐件留痕），新增 Flyway（先取号）。既有 approve 端点保持路径，语义升级为完整门禁，旧调用方语义不兼容处在契约文档标注。

## 风险

- custody 守恒破坏是 0.5 的 P0：以"领出=有效扫描且已批准"恒等式为硬校验，E2E 必测。
- 并发批准/部分失败：行锁+状态前置+事务回滚测试覆盖。

## 测试与 DoD

- Java：门禁矩阵、状态机、并发唯一成功、回滚无半截。
- 真实 MySQL E2E：建波次→两司机任务→扫描提交→门禁拦截→差异决定→批准→custody/事件/outbox 守恒断言→跨站 403；自清数据。
- Web：Vitest + Playwright；契约（approve 语义升级）、状态机文档、web-spec（4.6 刷新）中英同步。
