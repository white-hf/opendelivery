# O06 派送与回仓监督迭代

> 状态：`COMPLETED`（2026-07-21 完成）；归属：运营产品；对应双产品计划 O06。已关联 [执行总结](../summaries/iteration-o06-delivery-return-supervision-summary.md)。

## 可运营结果

运营跟踪当日在途、失败与退站：每张任务/司机的派送进度可见，失败件有明确下一动作（重派/退库/批准持有），包裹状态与 custody 始终守恒。已交付的失败退库接收是本迭代的事实基础之一。

## 范围

1. **在途监控**：`GET /ops/v1/delivery-monitor?serviceDate=` 按任务/司机聚合：应派、在途（OUT_FOR_DELIVERY）、妥投（DELIVERED）、失败（DELIVERY_FAILED）、回站；时效（首次尝试时间、最近事件）；有坐标的投递点在地图标记。
2. **失败队列**：失败原因分布；待重派（未超 retry 上限）、待退库（沿用已交付清单）、超限待处置三个分栏。
3. **批准持有**：对当日无法交回的失败件，主管可批准司机暂存并注明原因，计入守恒公式（领出=妥投+交回+批准持有）；批准留审计。
4. **同站重派**：失败件次日（或当日另一波次）重派到正确司机任务（复用改派事务），生成新 task item，原任务项置 REASSIGNED；不改站（跨城地址问题走 Case）。
5. **前端**：派送监控页（在途地图+任务列表、失败队列、批准持有/重派动作），三语言；导航"派送监控"进入（替换现退库单一页为分栏工作台，保留退库确认）。

## 非目标

- 不改 Driver 派送与 `/delivery/retry` 契约；不做实时轨迹与路线优化（I14）；不做 RETURN session（D04）。
- 不做跨站调拨与站间转运。

## 依赖

- `delivery_attempt`/`proof_of_delivery`/退库主链（V7 + 退库接收切片）、O05 custody 语义、O04 同源数字。

## 迁移与兼容

- 批准持有如落新表/新列（task item 状态或持有批准记录），新增 Flyway（先取号）；既有退库端点不变。

## 风险

- 守恒公式失真（P0）：E2E 断言"领出=妥投+交回+批准持有"为硬门禁，任何新动作必须先过公式。
- 重派与在途并发：task item 唯一活动槽约束（`uk_parcel_active_task`）兜底，冲突 409。

## 测试与 DoD

- Java：守恒公式、重派占用冲突、批准持有审计。
- 真实 MySQL E2E：造任务→在途→失败→批准持有/退库/重派三路径→公式断言→跨站 403；自清。
- Web：Vitest + Playwright；契约与 web-spec 中英同步。
