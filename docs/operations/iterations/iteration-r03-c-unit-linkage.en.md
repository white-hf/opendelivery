# R03-C Arrival Linkage Close-out Iteration (v2)

> Status: `REVIEWED` (approved 2026-07-21); product: Operations; closes O03 (arrival linkage) in the two-product execution plan. v2 revises the slice around the real station work flow: the arrival batch becomes the operational spine, and area-driven auto-fill becomes the primary unit-linkage channel.

## Work-flow position

Real station timeline: upstream pushes orders (possibly several times) → system routing and area-membership computation → the station plans the arrival batch and unit loading (area-driven auto-linkage with map review) → the upstream warehouse sorts and packs against the plan, truck arrives → operator confirms arrival → drivers scan and sort. Dispatch planning runs in parallel with arrival planning (efficiency work belongs to R04). This iteration owns the "batch / unit / parcel linkage and observation" part.

## Scope

1. **Arrival batch**: `arrival_trip` is the batch and `external_trip_no` is the batch number; it is auto-generated by default (`{stationCode}-{yyyyMMdd}-{two-digit seq}`) and can be overridden manually. The batch number is the single identifier of the run, reused by dispatch-plan codes (delivered in R04).
2. **Default units**: creating a batch also generates 10 PALLET units (`{batchNo}-U01..U10`); more can be added, so operators never key in ten units by hand.
3. **Automatic area membership**: after successful routing, ingestion computes `parcel_area_assignment` (order per PRD: manual override > valid Polygon/MultiPolygon > postal/city fallback > unassigned exception); adds a bulk recompute endpoint `POST /ops/v1/parcels/area-recompute` (defaults to the station's unmatched parcels of the day).
4. **Area auto-fill**: `POST /ops/v1/handling-units/{unitId}/area-fill {areaVersionIds[], reason?}` links every parcel currently in the selected areas to the unit with `link_source='AREA_PLAN'`; cross-station or unpublished areas are rejected with 409. Map picks and typed tracking numbers (`OPERATOR`) remain as occasional supplements.
5. **Upstream unit identifiers**: the ingestion request accepts an optional `handlingUnits` list (`externalUnitNo` required, optional `unitType`, per-unit `trackingNumbers`); `parcel.upstream_unit_no` (V13) stores the claim; parcels auto-link to same-station units with the same label both when a unit is created and during ingestion (`link_source='UPSTREAM'`).
6. **Coverage observation**: each unit in the trip detail returns `declared/linked/scanned/exception_piece_count`, `driver_count`, `wave_count`, parcel-level detail (tracking no / task / driver / item status / `link_source`), and the unlinked-declaration list; **aggregates always equal the detail rollup**. Exception definition: upstream-declared parcels that could not be linked to the unit (cross-station or unrouted); DAMAGED observations fold into the count once D02 scan classification lands.
7. **Three-city fixture**: YHZ/YYZ/YVR each with 2+ drivers, published areas and area-matched parcels; a batch with 10 default units; area fill produces one unit spanning two drivers' tasks and one task spanning two units; scripted checks assert aggregate=detail, cross-station isolation, and unknown-tracking rejection.
8. **Frontend workbench**: batch-centric layout following the work flow (create batch → default units → pick areas to auto-fill → map review → confirm arrival), coverage counters with drill-down, three languages.

## Non-goals

- No scanning on behalf of drivers; arrival facts never become parcel status or custody changes (D02/O05).
- Upstream declarations never auto-create trips or units; physical registration stays an operator action.
- No out-of-order handling of upstream unit amendments/cancellations (I10); no unlink UI (later iteration).
- No dispatch-planning efficiency work (auto plan codes, default capacity, driver default area assignment — that is R04).

## Dependencies

- Existing V12 arrival tables, the V8 area model (`parcel_area_assignment`, `delivery_area_version`), and the `driver_task_item` / `scan_event` main chain.
- `trackingNumbers` remains the parcel truth; disagreement with unit declarations only affects linkage and exception counts, never ingestion.

## Migration and compatibility (V13)

- `parcel` gains `upstream_unit_no VARCHAR(100) NULL` (index `(upstream_unit_no, current_station_id)`, serving both global label aggregates and station lookups) and `current_area_version_id BIGINT NULL` (denormalized projection of the active area assignment, index `(current_station_id, current_area_version_id)`; `parcel_area_assignment` remains the membership history, with backfill in the migration).
- The `ck_unit_parcel_source` CHECK on `handling_unit_parcel` is extended with `'AREA_PLAN'`.
- Only V13 is added; historical migrations are never edited. `handlingUnits` is optional and legacy payloads behave unchanged; a defaulted auto-generated `external_trip_no` does not affect clients that pass their own number.

## Risks

- Areas unpublished or membership not computed → area-fill returning empty must be surfaced clearly, with manual supplement as the path; unassigned-exception parcels follow the existing exception channel.
- Upstream labels or area membership disagreeing with physical freight → linkage is observation only; the operator's physical count wins; inventory and custody are untouched.
- Same label reused across stations → linkage is strictly station-scoped, covered by fixture and E2E.
