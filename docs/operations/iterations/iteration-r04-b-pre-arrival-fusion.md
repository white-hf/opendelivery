# R04-B 到仓与派送规划深度融合迭代

> 状态：`REVIEWED`（2026-07-22 评审通过）；归属：运营产品；承接 R03-C 到仓板笼模型与 R04 派送效率规划，将到仓规划与派送规划在【第 3 步：派送规划】深度融合，彻底理顺物理时序。

## 1. 背景与业界流程一致性 (Background & Industry Alignment)

### 物理世界时序倒挂痛点：
目前系统的【第 4 步：到仓接收】包含了“手动创建到仓批次”、“手动新增 Pallet”等操作。然而在实际物流场景中，**“干线板笼划分（Inbound Packing Plan）”必须在卡车开动甚至在上游分拣机启动前就绪（数据先行）**。
如果等到卡车到达站点（第 4 步）才在系统里人工建板笼，上游中心仓将无法获取区域映射规则进行物理预打包，流程存在严重的时序倒挂。

### 优化方案：
将**“干线板笼预规划 (Inbound Plan)”**与**“司机包裹指派 (Outbound Plan)”**深度并轨，统一移入【第 3 步：派送规划】（Pre-arrival Planning）。【第 4 步：到仓接收】则改造成纯粹的物理执行视图，仅保留卡车到港确认与实扫核对。

---

## 2. 迭代范围 (Scope)

### ① 统一骨架初始化 (Unified Backbone Initiation)
* **无感自动创建**：在【第 3 步】创建或选择派送波次（Wave）时，系统自动在后台静默生成同名的「到仓批次 (Arrival Trip)」，并预设 10 个 PALLET 类型的物理板笼单元（`U01~U10`），消除重复的人工建批次、建板操作。
* **计划编码一致性**：到仓批次号、派送计划编码、波次号共享同一串标示（`{stationCode}-{yyyyMMdd}-{seq}`），无感联结。

### ② 【第 3 步：派送规划】双通道并轨控制台 (Dual-Channel Sidebar Panel)
* **Tab 1：🚚 司机派车指派 (Outbound Plan)**：
  * 保留极简司机指派功能：`[ ⚡ 自动指派 ]`（按区域默认承包司机一键指派全部包裹）、司机下拉单号（显示已分件数）与地图局部联动（选定司机后，地图仅保留该司机已分件和未分配蓝件，隐藏其他司机干扰件）。
* **Tab 2：📦 干线板笼规划 (Inbound Plan)**：
  * **一键预装载 (Auto area-fill)**：一键调用后台 `/ops/v1/handling-units/{unitId}/area-fill`，按物理区域默认配置，将全站包裹自动装入 `U01~U10` 各个板笼中。
  * **板笼列表与覆盖观察**：直观展示板笼清单、容量填充比例（如 `板笼 U01: 45 / 100 件`）和绑定的区域多选下拉框。
  * **地图联动高亮**：在右侧点击选中某板笼，左侧地图瞬间高亮其负责的物理配送小区（Area Polygon）和包裹点，并以紫色边框标示。

### ③ 【第 4 步：到仓接收】纯执行化重构 (Simplification of Step 4)
* **移除冗余表单**：彻底砍掉“创建到仓批次号”、“手动新增 Pallet”等一切前置规划表单。
* **极简到港确认**：操作员进入页面时，直接展示在「第 3 步」已经生成好的对应批次和板笼清单。只需一键点击 **`[ 🚚 确认卡车到港 / 确认到仓 ]`**（调用底层 `PUT /ops/v1/trips/{tripId}/arrive`），批次变为 `ARRIVED` 状态，立刻放行扫码清点。

---

## 3. 非目标 (Non-Goals)

* 本迭代专注于 UI 交互时序重构与前后端接口串联，不改动底层的 `handling_unit_parcel`、`arrival_trip` 数据库实体关系（保持 V12/V13 架构）。
* 不改动司机 App (Driver App) 的 `LOAD` 物理交接门禁逻辑。

---

## 4. 依赖 (Dependencies)

* 依赖 `V12`、`V13` 物理到仓和上游预装载数据契约。
* 依赖后台现有的 `/ops/v1/handling-units/{unitId}/area-fill` 接口。

---

## 5. 迁移与兼容 (Migration & Compatibility)

* **向下兼容**：原有的 API 接口和入参校验全部保持兼容。波次和批次创建底层自动咬合，旧的手工创建 API 保留为可选通道，确保现有单元测试和集成测试顺利通过。

---

## 6. 测试与 DoD (Testing & Definition of Done)

* **DoD 1**：前台【第 3 步】新增 Tab 控制，板笼与司机指派并轨，完全摆脱多余说明文字。
* **DoD 2**：前台【第 4 步】移除所有手动创建 TRIP 和板笼的输入框，进入页面可直接加载并确认到港。
* **DoD 3**：整个前端服务运行 `npm run build` 打包无任何 TypeScript 编译错误。
* **DoD 4**：后端服务运行 `./run.sh test` 保证所有既有测试完美通过。
* **DoD 5**：双语（中文及英文）产品规划文档归档，并在控制台以规范、精炼的术语呈现。
