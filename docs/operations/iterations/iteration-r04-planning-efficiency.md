# R04 派送规划效率迭代

> 状态：`COMPLETED`（2026-07-21 完成）；归属：运营产品；承接 R03-C 的到仓批次，按站点实际作业流消除规划环节的重复人工操作。已关联 [执行总结](../summaries/iteration-r04-planning-efficiency-summary.md)。

## 背景与现状差距

站点作业流要求：派送规划在卡车到达前、上游首次推送后即可进行，多次推送可增量处理；计划编码统一复用到仓批次号；运力默认可用、仅登记例外；司机按绑定区域（1-N）默认分得本批次包裹，地图仅偶尔调整。

现状差距（代码实证）：`driver_shift` 无行即视为 `UNAVAILABLE` 且分配直接报错——每司机每日都要建档；`waveCode` 手工必填；司机分配每次都要地图手选（API 虽支持 `WHOLE_AREA`，但无按司机偏好区域的默认分配）；上游多次推送后的增量包裹没有待规划视图。

## 范围

1. **运力默认可用**：无 `driver_shift` 行视为 AVAILABLE，默认容量取站点级配置；仅请假、生病等例外建档（`UNAVAILABLE` 及原因）。班次界面从"逐人每日建档"改为"例外标记"。
2. **计划编码自动生成**：`WaveRequest.waveCode` 降为可选；缺省 `{arrivalBatchNo}-W{seq}`（无批次时 `W{yyyyMMdd}-{seq}`），与到仓批次保持统一标识；人工可覆盖。
3. **司机默认分配**：按 `driver_area_preference`（司机绑定 1-N 区域）把本批次区域内未分配包裹默认挂入司机任务（`WHOLE_AREA`），容量与占用校验沿用；地图 `reassign` 调整保留。
4. **增量处理**：上游多次推送后，未规划包裹进入"待规划"视图（按区域聚合）；可对增量再次默认分配或手工调整，不影响已发布任务。
5. **规划工作台 UI**：按"待规划 → 默认分配 → 地图微调 → 冻结/发布"作业流重构布局，减少点击与跳转；三语言。

## 非目标

- 不做路线优化与 stop sequence 建议（I14 范围）。
- 不改发布门禁与交接审批（O05）；Driver API 契约不变。
- 不做多波次自动拆分或合单。

## 依赖

- R03-C 的到仓批次号与自动区域归属；V8 `driver_area_preference`、V11 波次/任务主链。

## 迁移与兼容

- 站点级默认容量配置如需落库，新增 Flyway（只增不改）。
- `waveCode` 由必填降为可选；旧调用方显式传码行为不变。

## 风险

- 默认可用导致请假司机被排入 → 例外标记入口置于规划页显著位置，默认分配前重新校验当日例外。
- 默认分配与实际需求偏差 → 发布前地图核对仍是必需步骤，审批门禁不变。

- Java：无 shift 默认可用、例外拦截、编码自动生成且同站唯一、默认分配的容量/跨站/占用校验、增量再分配幂等。
- 真实 MySQL 与三城市 fixture 验证默认分配结果；Web：Vitest + Playwright 规划工作台回归；中英文档与执行总结按治理流程交付。
