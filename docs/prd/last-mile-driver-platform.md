# OpenDelivery Last-Mile Driver Platform PRD

> 本文保留为早期需求背景。当前产品和运营权威规格请使用 [产品与运营一体化规格](product-and-operations-specification.md)，避免将早期目标与 MOV 承诺混淆。

## 1. 背景与目标

OpenDelivery 是末端配送团队的司机 App 后台。包裹主数据来自上游客户/承运商通过 API 推送或定时拉取，本系统负责接收、校验、站点入库、分拣派车、司机交接、末端派送、POD、异常处理及结果回传。目标是让每票包裹从上游接入到交付/退回形成可追踪、可审计、可重放的闭环。

当前实现仅供演示：`MemoryDataStore` 内置包裹并直接修改状态；没有上游接入、站点库存、正式派车、持久化、完整状态历史或可靠回传。

## 2. 角色与系统边界

- **上游客户系统**：提供运单、收件信息、服务要求和取消/更址等变更；接收状态与 POD 回传。
- **集成服务**：处理推送、拉取、验签、幂等、字段映射、批次对账和重试。
- **站点运营人员**：接收入库、分拣、建派送波次、分配线路和司机、处理差异。
- **司机**：登录、查看任务、装车扫描、派送、提交成功/失败及 POD、退件交接。
- **客服/调度/主管**：改派、重约、异常处理、批次审核和运营查询。
- **OpenDelivery**：是末端执行过程的事实来源；上游仍是商业运单和客户指令的来源。

## 3. 核心业务对象与批次来源

必须区分四类“批次”：

1. **接入批次（ingestion batch）**：一次上游推送请求、文件或拉取分页任务产生；记录来源、外部批次号、原始载荷、成功/失败数量和对账结果。
2. **到站清单（inbound manifest）**：上游声明将到达某站点的一组运单；可跨多个接入批次更新，但以 `upstream + external_waybill_no` 唯一识别运单。
3. **派送波次（dispatch wave）**：站点基于服务日期、区域、线路、容量和时窗生成的派送集合，并分配司机/车辆。
4. **司机扫描批次（scan session）**：司机在交接/装车时开启的操作会话，用于记录应扫、实扫、多扫、少扫及审核结果；它不拥有运单主数据。

## 4. 端到端闭环

1. **接入**：上游推送或系统按游标拉取运单；保存请求和幂等键，校验客户、站点、地址、时窗和必填字段。
2. **落单与变更**：合法数据创建/更新运单与包裹；非法数据进入隔离队列并返回逐条错误。已进入不可变阶段的更址/取消转人工处理。
3. **到站与分拣**：实物到站扫描后进入站点库存；分拣将包裹关联服务区、线路和计划配送日，差异进入少货、多货或错站异常。
4. **计划与派车**：调度创建派送波次，生成司机任务及任务明细；只有站点已持有且可派送的包裹可以分配。
5. **司机交接**：司机打开扫描批次，对任务应扫清单装车扫描。全部匹配或主管接受差异后完成交接，包裹转为 `OUT_FOR_DELIVERY`。
6. **末端执行**：司机到达后提交成功或失败。成功必须满足 POD 策略；失败必须选择原因，可生成重约、改派或退回流程。
7. **闭环回传**：每次有效状态变化写入不可变事件，并进入 outbox；集成服务按上游协议回传，失败重试并可人工重放。上游确认后记录回执。
8. **日终对账**：对接入数量、站点库存、司机持有、已交付、异常和待回传数量做平衡检查，未闭环项目进入运营工作台。

运营闭环不仅依赖包裹状态，还必须同步维护物理责任（custody）、库存位置、任务、异常工单和上游回执。班前准备、到站差异、波次发布门禁、司机双向交接、动态改派、收车退件和主管日终签字的详细要求见 `operations-closure-review.md`。

## 5. 包裹生命周期

主路径：

`RECEIVED → AT_STATION → SORTED → READY_FOR_DISPATCH → ASSIGNED → OUT_FOR_DELIVERY → DELIVERED`

异常路径：

- `OUT_FOR_DELIVERY → DELIVERY_FAILED → RESCHEDULED → ASSIGNED`
- `DELIVERY_FAILED → RETURN_PENDING → RETURNED_TO_STATION → RETURNED_TO_UPSTREAM`
- 可在允许阶段进入 `CANCELLED`；实物已在途时必须先回站。
- `LOST`、`DAMAGED`、`ADDRESS_EXCEPTION` 为受控异常状态，需要原因、操作者和审核记录。

状态只能由领域命令改变，必须校验前置状态并记录 `parcel_status_event`。扫描事件本身是事实记录，只有满足业务规则后才触发状态变化。

## 6. 功能范围

- 上游租户、凭证、推送/拉取配置、映射、游标、接入批次、隔离和重放。
- 运单、包裹、地址、服务时窗、客户指令、标签及变更版本。
- 站点、服务区、线路、司机、车辆、班次、派送波次及任务分配。
- 到站、分拣、装车、卸货、交付、退回等扫码及差异审核。
- 司机认证、任务列表、离线幂等提交、位置点、POD、失败原因和重试。
- 改派、重约、取消、丢损、错站、拒收和退回等异常工单。
- 状态事件、上游回传、重试、死信、回执、日终对账和审计查询。
- 运营指标：准时率、首次妥投率、扫描差异率、积压、回传延迟及失败率。
- 班前健康检查、站点/司机工作台、在途监控、异常 SLA 队列、交接签字和日终关站清单。

## 7. 关键业务规则与验收

- 上游重复推送不得重复建单；同一来源的外部运单号唯一，更新按外部版本或事件时间有序处理。
- 包裹同一时刻最多属于一个有效派送任务；交接完成前不能进入派送中。
- POD 规则按客户/服务类型配置；图片对象存对象存储，数据库只保存元数据与校验值。
- 所有写操作支持请求幂等键，状态更新采用版本号防止司机与调度并发覆盖。
- 每个终态均能追溯来源载荷、每次状态变化、持有人、POD、回传尝试和上游确认。

## 8. 待业务确认

- 首批上游协议、字段样例、推送鉴权方式与拉取频率。
- 是否一票多件，以及运单级和包裹级状态的聚合规则。
- 站点入库是否强制实物扫描，司机是否允许差异装车。
- POD 类型、失败原因、重试上限、退回 SLA 和敏感数据保留周期。
- 路线规划由上游提供、运营手工维护，还是后续接入优化引擎。
