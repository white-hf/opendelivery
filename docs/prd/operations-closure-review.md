# Last Mile 运营闭环评审

> 本文保留为早期评审证据。当前闭环、功能方案与验收已合并到 [产品与运营一体化规格](product-and-operations-specification.md) 和 [MOV 必备功能解决方案](mandatory-capability-solutions.md)。

## 1. 评审结论

现有 PRD 已覆盖“上游接入—司机扫描—派送—结果回传”的主路径，但从运营人员每天开站、接货、派车、盯盘、收车和关站的视角，原流程仍缺少六类闭环：主数据开通、实物与系统对账、任务发布前门禁、司机交接责任、异常工作台、日终关账。仅有包裹状态不足以运营，必须同时管理“库存位置、责任持有人、任务、异常工单、上游回执”五条事实链。

## 2. 运营日完整流程

### 2.1 开通与班前准备

客户经理/系统管理员先开通上游伙伴、站点、服务区、服务产品、POD 策略、失败原因、回传协议和 SLA。站点主管维护司机、车辆、班次、线路容量及当日营业日历。系统执行连接器健康检查、昨日未闭环项检查和当日预计到货预报。

**门禁**：配置未生效、上游连接失败、站点停用或司机/车辆资质无效时，不允许发布相关任务，并生成待办。

### 2.2 预报、接货与到站对账

上游预报形成 inbound manifest；车辆到站后，操作员登记到达时间、封签/笼车/袋号并开始卸货扫描。系统将实扫与预报比较，产生正常、少货、多货、重复、错站、条码不可读和数据未到等差异。

**人工参与**：站点主管处理差异。多货可等待上游补数或创建隔离记录；少货不能进入站点可用库存；错站生成转运任务。manifest 只有在差异接受或解决后才能关闭。

### 2.3 分拣、计划与波次发布

系统按地址质量、邮编/服务区、时窗、服务产品、路线和容量给出分拣/派车建议。调度员处理无区域、地址异常、超容量和特殊件，然后创建并冻结派送波次。

**发布门禁**：包裹必须实物在站、数据完整、未取消、未被其他有效任务占用；司机在岗且容量允许。发布后变更必须走撤回/改派，不能静默覆盖司机任务。

### 2.4 司机签到、装车与责任交接

司机签到并确认车辆/设备。司机针对已发布任务开启 scan session，扫描装车。系统实时显示应扫、实扫、漏扫、多扫、串线和重复扫描。无差异时司机确认接收；有差异时主管决定补扫、移除、接受差异或取消发布。

**责任转移**：只有双方确认后，包裹 custody 才从站点转为司机，任务进入 `IN_PROGRESS`，包裹进入 `OUT_FOR_DELIVERY`。每次转移记录操作者、时间、设备和位置。

### 2.5 在途执行与动态调度

司机按任务顺序导航、联系收件人并提交到达/尝试结果。App 可离线记录命令，每条命令带设备事件 ID，恢复网络后幂等同步。客服或调度收到取消、更址、优先级变更时，系统根据当前 custody 决定自动生效或发起人工拦截。

**人工参与**：调度处理司机故障、超时、任务转派和路线调整；任何改派需要原司机交回或数字交接，避免双重持有。

### 2.6 妥投、失败、重约与退回

妥投按客户策略采集照片、签名、OTP、收件人关系、GPS 和时间；系统校验后完成交付。失败必须使用配置化原因码并采集必要证据，规则引擎决定可重试、客服确认、次日重约、退回站点或直接退上游。

**人工参与**：高价值、拒收、损坏、丢失、身份验证失败和超过重试上限进入异常工单，由有权限人员审批。COD、年龄限制和危险品不是默认能力，只有产品/合规规则确认后才能启用。

### 2.7 收车、退件交接与日终关站

司机结束班次前必须完成任务 closeout：未交付包裹逐件回站扫描，POD/离线事件全部同步，现金或设备等可选资产另行交接。站点接收后 custody 回到站点，并生成重约或退回任务。

系统执行日终平衡：期初库存 + 到站 + 转入 - 派出 + 退回 - 妥投 - 转出 = 期末库存。并核对司机持有、开放任务、未解决差异、未上传 POD、未确认回传和超龄库存。

**关站门禁**：所有不平项必须解决或由主管带原因签字结转；系统保留交班清单和次日待办。

### 2.8 上游回传与业务关单

每个可回传状态事件写入 outbox；连接器重试并记录上游 ACK。技术发送成功不等于业务闭环，只有上游接受或人工确认无需回传时才关闭。每天用上游状态快照/对账文件比对，差异形成 reconciliation case。

## 3. 必须新增的系统功能

| 能力 | 系统自动化 | 人工责任 |
|---|---|---|
| 配置中心 | 版本化伙伴、站点、服务、原因码和策略 | 管理员提交，业务负责人审批 |
| 到站工作台 | manifest 对账与差异分类 | 操作员扫描，主管关差异 |
| 调度工作台 | 容量校验、任务冲突、发布门禁 | 调度建波次、冻结、改派 |
| 交接工作台 | 应扫/实扫差异和 custody 事件 | 司机确认，主管接受例外 |
| 在途监控 | 超时、离线、偏离和积压告警 | 调度介入、联系司机或改派 |
| 异常工单 | SLA、队列、升级、证据和操作历史 | 客服/主管调查并决策 |
| 退回管理 | 回站、重约、退上游的任务链 | 站点与司机实物交接 |
| 日终对账 | 库存平衡、任务/POD/回传完整性 | 主管签字结转 |
| 上游回传台 | outbox、重试、死信、ACK、重放 | 集成运营处理长期失败 |
| 审计与权限 | RBAC、敏感字段脱敏、不可变审计 | 安全管理员定期复核 |

## 4. 关键运营状态必须分离

- **包裹生命周期**：业务进展，如 `AT_STATION`、`OUT_FOR_DELIVERY`、`DELIVERED`。
- **物理责任 custody**：`UPSTREAM`、`STATION`、`DRIVER`、`RETURN_CARRIER`。
- **库存位置**：站点、分拣区、笼车、车辆或未知。
- **任务状态**：`DRAFT`、`PUBLISHED`、`ACCEPTING`、`IN_PROGRESS`、`CLOSED`、`CANCELLED`。
- **异常工单状态**：`OPEN`、`ASSIGNED`、`WAITING_EXTERNAL`、`RESOLVED`、`CLOSED`。
- **回传状态**：`PENDING`、`SENDING`、`ACKNOWLEDGED`、`RETRY`、`DEAD_LETTER`、`WAIVED`。

这些状态互相关联但不能用一个字段替代。例如包裹可以处于 `DELIVERY_FAILED`，同时 custody 为 `DRIVER`，任务仍为 `IN_PROGRESS`，回传已 `ACKNOWLEDGED`。

## 5. 运营闭环验收标准

- 任一包裹能回答：数据从哪里来、实物在哪里、谁负责、下一动作是什么、是否超 SLA、上游是否确认。
- 任一批次能回答：期望多少、实际多少、差异多少、谁处理、何时关闭。
- 任一司机班次能回答：领走多少、妥投多少、失败多少、交回多少、是否仍有未同步事件。
- 任一异常均有 owner、优先级、截止时间、证据、决策和关闭原因。
- 任一业务终态均有状态事件、POD/原因、custody 终点及上游 ACK 或豁免。
- 站点可以在不查询数据库的情况下完成班前检查、派车、在途监控、收车和日终签字。

## 6. 本期实施边界

首个可生产闭环实现上游接入、运单/包裹、站点库存、波次/任务、装车交接、派送尝试、POD 元数据、异常、状态事件/outbox 和日终查询。路线优化、实时轨迹、COD 结算、计费、客户门户和复杂合规件作为后续迭代；数据模型预留扩展但不虚构规则。
