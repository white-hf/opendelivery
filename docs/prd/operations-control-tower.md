# 运营控制塔与流程导航产品规格

## 产品目标

运营系统必须让新运营人员在一分钟内回答：今天处理多少件、运力是否足够、当前在哪个阶段、什么正在阻塞、下一步去哪里处理。系统以“站点营业日”为工作上下文，导航表达业务流程，首页负责判断与引导，专业工作台负责执行。首期服务 YHZ、YYZ、YVR 一城一站；不建设总部视图、跨站转运和自动路线优化。

## 用户与工作模式

| 角色 | 首要关注 | 高频动作 |
|---|---|---|
| 调度员 | 订单分布、运力缺口、未分配包裹 | 配置班次、建批次、整区/拆分、冻结发布 |
| 到仓运营 | 卡车/板到达、扫描差异 | 登记到仓、看进度、处理错件/破损 |
| 站点主管 | 阻塞、责任风险、放行/关站条件 | 审批、改派、关闭异常、签字结转 |

站点和营业日是全局上下文；切换后指标、地图、列表、待办和缓存必须同时切换。

## 流程导航

左侧一级导航按实际营业日顺序排列，并显示待办数、阻塞数和当前推荐步骤：

1. **今日运营**：控制塔、阶段进度和下一步行动。
2. **订单准备**：接入结果、路由、地理编码、区域匹配和数据异常。
3. **派送规划**：地图、班次/容量、批次、分配、冻结和发布。
4. **到仓接收**：车次、板、实物到达与外观异常。
5. **司机扫描**：本人应扫清单、进度、漏扫/错扫/破损。
6. **交接审批**：批次核对、异常关闭和 custody 放行。
7. **派送监控**：派送中、妥投、失败、再次尝试和回仓。
8. **日终关站**：责任守恒、未结事项、结转和签字。
9. **配置**：区域、司机、站点和权限；与每日作业分组展示。

阶段由业务事实计算 `NOT_STARTED/IN_PROGRESS/BLOCKED/COMPLETED`，不能人工修改。点击阶段进入工作台并自动带入站点、营业日和阻塞筛选。

## 首页“今日运营”

顶部显示站点名称/编码、营业日、当地时间、数据更新时间和数据源状态。`stationId=3`、`ready=true` 等技术字段不得直接展示，必须转换为运营语言和具体阻塞原因。

主流程条展示“订单准备 → 派送规划 → 到仓 → 扫描 → 审批 → 派送 → 回仓 → 关站”，每步显示完成比例、待办/异常数并可点击钻取。

核心指标包括预计订单、已路由、已分区、已分配、已到仓、有效扫描、已放行、派送中、妥投、失败/回仓。运力卡展示出勤司机、总容量、已分配、剩余和缺口。异常卡展示未路由、缺坐标、未分区、超容量、未提交扫描、破损、待审批和未关闭 Case。所有数字必须可点击，聚合和明细使用同一服务端口径。

“下一步行动”按严重度、阶段和截止时间排序。例如“38 件未分配，进入地图规划”或“2 件破损阻塞交接审批”。每条必须说明数量、影响、建议动作并提供直达按钮。

## 地图工作区

地图、指标和列表共享查询/筛选。顶部明确显示“查询 72 件 / 可定位 69 件 / 缺坐标 3 件 / 地图显示 69 件”。首次加载自动缩放到全部点位，并提供“定位全部”和“清除筛选”。

点位按未分配、已分配、数据异常、运营异常使用固定颜色；密集区域聚合。点击指标、图例或待办同步过滤地图和列表。空地图必须区分没有订单、筛选无结果、全部缺坐标、Google Maps 失败和 API 失败，并给出恢复动作。

Google Maps 只接收经纬度和样式，不发送姓名、电话或完整地址；详情由本系统抽屉按需加载。

## 测试订单体验模式

上游 API 未接通期间，使用 `scripts/db/004_r02_experience_seed.sql` 模拟 Canonical 接入结果。数据统一使用 `DEMO-R02-*`；YHZ、YYZ、YVR 各包含 6 名司机、4 个发布区域和 72 个当日订单。

每站覆盖标准/加急、上游 custody/已到站、四个区域、缺坐标、区域外坐标和不同司机容量。体验数据必须走正式 Waybill、Parcel、Geocode、Area Assignment 和规划门禁，测试操作仍记录审计。

## 权限、多语言与验收

普通运营只能使用授权站点；管理员切站必须取消旧请求并清除旧站缓存。导航、指标、状态、异常和行动建议支持 `en-CA`、`fr-CA`、`zh-CN`。

- 新运营人员 60 秒内找出当前阶段、最高优先阻塞和下一动作。
- 首页所有数字可钻取，聚合与明细数量一致。
- 有坐标测试订单时，地图首屏必须出现点位且显示数与列表一致。
- 三站切换后无旧站数字、点位、列表或待办残留。
- 核心流程可仅按导航顺序完成，不依赖培训人员解释菜单。
- 自动化和运营浏览器验收均通过后才能标记完成。
