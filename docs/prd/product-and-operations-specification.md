# OpenDelivery 产品与运营一体化规格

> **已被替代：** 标准运营主流程以 [运营系统产品模型](operations-system-product-model.md) 为准。本文件仅保留早期需求历史。

## 1. 产品定位与首要目标

OpenDelivery 是一个 Last Mile 履约平台，包含**司机作业子系统**和**运营管理子系统**，由上游集成、站点路由与核心履约能力支撑。首要目标不是一次达到行业标杆，而是让多个城市各自的单一站点可以不用查库完成每天的接货、派车、交接、配送、异常处理、回传和关站。

首个最小可运营版本（MOV）限定为：同一部署支持多个城市、每个城市一个站点、所有站点使用统一业务时区规则、上游使用 Canonical JSON 提供地址与服务要求、系统按版本化服务范围规则选择站点、运营人员人工建波次、司机 App 执行。组织层级、一个城市多个站点、站间转运、自动路线优化、实时 GPS、COD、计费、客户门户和 SFTP/PULL 不进入 MOV。

## 2. 用户、职责与系统边界

| 角色 | MOV 职责 | 使用入口 |
|---|---|---|
| 上游系统 | 推送地址、服务与包裹数据，接收状态回调；无需了解内部站点 | Integration API |
| 入站操作员 | 卸货扫描、标记差异 | 运营系统 |
| 调度员 | 选包裹、建波次、分配司机 | 运营系统 |
| 站点主管 | 审核差异、交接例外、日终签字 | 运营系统 |
| 司机 | 接单、装车扫描、配送、失败回站 | Driver API / App |
| 客服/异常专员 | 处理地址、拒收、破损等工单 | 运营系统 |
| 集成运营 | 处理回调死信和上游对账差异 | 运营系统 |

上游是商业运单及客户指令的来源；OpenDelivery 是站点收到实物之后的库存、责任交接和末端执行事实来源。

## 3. 业务对象与批次

- **Waybill**：上游商业运单，可包含多个 Parcel。
- **Parcel**：带唯一 tracking number 的可扫描物理件。
- **Ingestion Batch**：一次推送、拉取页或文件的技术接入批次。
- **Inbound Manifest**：上游声明将到达站点的包裹清单。
- **Dispatch Wave**：站点按服务日期/线路组织的派送计划。
- **Driver Task**：波次分配给一名司机的执行任务。
- **Scan Session**：装车、回站或转交的一次扫描会话。
- **Delivery Attempt**：一次成功或失败的派送尝试。
- **Operational Case**：不能自动闭环、需要明确 owner 和 SLA 的异常。

这些对象不可互相替代。例如扫描批次不产生运单，司机任务也不代表物理责任已经交给司机。

## 4. 最小可运营闭环

```mermaid
flowchart LR
    A[上游推送地址与运单] --> B[校验与幂等落单]
    B --> B1[地址标准化与站点路由]
    B1 --> C[按结果生成到站预报]
    C --> D[操作员实物收货扫描]
    D --> E{预报与实扫一致?}
    E -- 否 --> F[异常工单/主管处理]
    F --> G[可派送库存]
    E -- 是 --> G
    G --> H[调度建波次并分配司机]
    H --> I[司机装车扫描]
    I --> J{交接差异?}
    J -- 是 --> K[主管接受或退回调整]
    K --> L[责任转移给司机]
    J -- 否 --> L
    L --> M{派送结果}
    M -- 成功 --> N[POD校验并妥投]
    M -- 失败 --> O[失败原因/回站或重派]
    O --> P[司机回站扫描交回]
    N --> Q[状态回传]
    P --> Q
    Q --> R[日终平衡与主管签字]
```

### 4.1 上游接入

上游以 `partnerCode + externalEventId` 幂等推送运单，必须提供国家、省、城市、邮编、地址行和服务要求，不必提供内部站点。系统先记录原始请求摘要，再校验 Partner、地址、时间窗和 tracking number；Routing 模块标准化地址，按有效服务范围规则唯一选择站点。成功后创建 Waybill、Parcel 和该站点 Manifest；无匹配、多匹配或地址不足时 Waybill 进入路由异常 Case，不进入可派库存。MOV 只承诺 Canonical JSON；其他格式通过后续 Partner Adapter 接入。

### 4.2 到站接货

操作员按 Manifest 扫描实物。匹配包裹进入 `AT_STATION`，custody 转为 `STATION`；少货、多货、错站、破损或无数据形成差异。少货不能进入可用库存，多货必须等上游补数或主管隔离。Manifest 差异解决或签字接受后才能关闭。

### 4.3 波次与派车

调度从站点可派送库存选择包裹，指定服务日、线路和司机。发布门禁：包裹实物在站、数据完整、未取消、没有其他有效任务；司机有效且属于本站。MOV 允许人工选择，不做自动路线优化。

### 4.4 装车与责任交接

司机开启 LOAD Scan Session，逐件扫描。系统显示应扫、实扫、漏扫、多扫和错任务。无差异时司机提交；有差异时主管补扫、移除或带原因接受。批准后任务进入 `IN_PROGRESS`，Parcel 进入 `OUT_FOR_DELIVERY`，custody 从站点转为司机。

### 4.5 配送、失败与回站

司机只能操作本人有效任务。成功派送至少提交位置、时间、收件人和照片 POD；失败必须提交原因。MOV 支持失败后回站或由运营次日重新建任务，不做自动重约。司机收车时逐件 RETURN 扫描，站点确认后 custody 回站，未解决件进入异常队列。

### 4.6 回传与日终

状态变化、事件和 outbox 在同一事务提交。回调失败自动重试，超过阈值进入死信；集成运营可重放。每天核对：期初库存 + 到站 - 派出 + 退回 - 妥投 = 期末库存，同时检查司机持有、开放任务、POD、异常和待回传。所有不平项必须解决或由主管说明后结转。

## 5. 独立状态与关键规则

- Parcel lifecycle、custody、物理位置、task、case、callback 是六套独立状态。
- 每个状态命令必须校验前置状态、操作者权限、幂等键和版本号。
- 同一 Parcel 同时最多处于一个有效 Driver Task。
- Waybill 在进入站点库存前必须为 `ROUTED`；Parcel、Manifest、Wave、Task、Driver 的站点必须一致。
- 包裹一旦实物到站，不允许系统自动改站；地址变更进入人工 Case。MOV 不支持站间转运。
- 物理责任只通过双方确认的 Custody Event 转移。
- 终态必须具备状态事件、成功 POD 或失败原因，以及上游 ACK/豁免。
- 任一异常必须有 owner、优先级、SLA、证据、动作历史和关闭原因。

## 6. MOV 功能清单与现状差距

| 能力 | 当前状态 | MOV 解决方案/验收 |
|---|---|---|
| Canonical 上游推送 | 已实现基础版 | 移除上游站点必填、Partner 凭证、逐条校验、原始报文和重放 |
| 地址标准化与站点路由 | 未实现 | 服务范围配置、确定性 Routing Service、Waybill 当前结果、异常 Case |
| 多城市一城一站 | 数据基础 | 城市属性、站点上下文、全链路一致性校验、按站点独立运营 |
| Manifest 收货 | 部分实现 | 工作台、差异分类、关闭门禁 |
| 波次/任务 | 部分实现 | 草稿、校验、发布、撤回；禁止重复分配 |
| 装车交接 | 部分实现 | 司机提交 + 主管审批 + custody 原子转移 |
| 配送/POD | 已实现基础版 | POD 策略、文件限制、授权查看 |
| 失败/回站 | 未闭环 | 原因码、RETURN session、回站确认、次日重派 |
| 异常工单 | 数据基础 | 队列、owner、SLA、动作和关闭接口 |
| 回传工作台 | 后台重试已实现 | 查询、死信、重放、ACK 状态 |
| 日终关站 | 数据基础 | 平衡计算、不平项、主管签字和结转 |
| 运营身份权限 | 未实现 | 用户、角色、站点权限和审计，不再使用全局 API Key |

详细技术方案见 [MOV 必备功能方案](mandatory-capability-solutions.md)。

## 7. 运营验收与成功指标

MOV 上线必须满足：至少三个城市、每城一个站点在同一部署中连续 5 个营业日完成独立闭环；上游不传内部站点也可完成路由；任一跨站点混用被拒绝；每件包裹能回答来源、站点、位置、责任人、下一动作、SLA 和回传状态；账实差异全部进入工单；司机任务无越权和重复持有；状态回传最终成功率 ≥99.5%；各站点日终不平项有负责人和处理结论。

观察指标包括接入接受率、Manifest 差异率、装车差异率、首次妥投率、失败原因分布、司机未交回数、POD 缺失数、回传延迟和超 SLA 工单数。首轮试运营只建立基线，不设置脱离真实数据的行业标杆目标。

## 8. 暂不纳入 MOV

自动路线优化、实时轨迹、动态 ETA、客户自助门户、COD/结算、司机计酬、车辆资产、复杂危险品/年龄验证、SFTP/PULL、多站点调拨、数据仓库和高级经营分析进入后续版本。
