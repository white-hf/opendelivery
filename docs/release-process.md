# 发布与回滚流程

## 发布前

冻结需求与 migration，完成代码评审、全量测试、数据库备份和空库 migration dry-run。确认上游回调、对象存储、告警、API Key、JWT 和数据库凭据由 Secret 管理提供。核对开放异常、outbox 积压和站点业务窗口。

## 发布

先部署兼容数据库 migration，再滚动发布应用。首批仅开放测试伙伴/站点，观察登录、接入拒绝率、任务查询、状态转换、数据库连接池、outbox 延迟及 5xx。达到稳定窗口后扩大范围。

## 回滚

应用回滚到上一兼容版本；Flyway migration 默认前向修复，不自动 DROP。若 migration 不向后兼容，发布前必须提供明确的补偿脚本并经备份验证。暂停上游接入时保留原始请求和游标，恢复后幂等重放。POD 文件和领域事件不得因应用回滚删除。

## 停止条件

出现数据不一致、重复妥投、跨司机泄露、migration 错误、持续回调失败或高比例 5xx 时停止推广并回滚。发布负责人记录时间线、影响、处置和后续行动。
