# OpenDelivery 产品版本与迭代路线图

## 1. 当前状态

| 版本/迭代 | 状态 | 已有结果 |
|---|---|---|
| I00 Domain Baseline | 已完成 | 核心领域、状态机、数据库基线 |
| I01 Persistence Foundation | 已完成 | MySQL/JDBC、Flyway、基础接入、司机主链路、POD、Outbox、真实 MySQL E2E |
| `0.2 Foundation` | 已完成 | 可持久化演示，但不能正式运营 |
| I02–I06 后端 | 已完成 | 路由、身份、入站、调度交接、失败回站；运营 Web/系统自动化待补 |

| I09–I15 | 暂停 | 通过 `0.5` Gate 后才重新排序 |

## 2. 规划原则与目标边界

版本以可验证的运营结果为边界，每个迭代原则上为一周并交付一个纵向切片。首要目标是同一部署支持多个城市、每个城市一个站点，完成系统自主路由和逐站点闭环；不建设组织层级、一个城市多个站点、站间转运、多时区关站或总部汇总。

## 3. 版本总览

| 版本 | 目标 | 主要范围 | 发布证据 |
|---|---|---|---|
| `0.2 Foundation` | 建立持久化基础 | MySQL、Flyway、核心实体、幂等、Outbox、基础 E2E | 已完成 |
| `0.5 Multi-City MOV` | 多城市、一城一站可实际运营 | 系统路由、运营 Web/RBAC、入站、波次、交接、失败回站、Case、回传、逐站点关站 | 至少3城市连续5天 |
| `1.0 Stable Operations` | 稳定服务首批真实上游 | 异构 Partner Adapter、变更/取消、生产安全、监控 SLO、容量、备份恢复 | 契约、容量、灾备和安全门禁 |
| `1.1 Operational Efficiency` | 降低站点人工成本 | 批量操作、基础顺序建议、通知、SLA 看板、运营指标 | 操作时长/积压有实测改善 |
| `2.0` | 暂不承诺范围 | 只有 `1.1` 真实运营证明需要后再规划 | 新 PRD 与架构评审 |

## 4. `0.5 Multi-City MOV`：I02–I08（承诺计划）

| 迭代 | 可演示结果 | 核心范围 | 完成标准 |
|---|---|---|---|
| I02 多城市路由基础 | 上游不传站点，系统把三个城市运单分到正确站点 | station 城市字段、一城一站、service-area、Routing Service、Waybill 当前结果、路由 Case、站点选择器 | 正常/无匹配/多匹配/人工覆盖及跨站点拒绝 E2E |
| I03 运营身份与准备 | 运营用户按角色进入默认站点，管理员可切站 | Operator Token、角色、默认站点、审计、班前检查、运营 Web 壳 | 权限矩阵和切站缓存测试 |
| I04 入站闭环 | 各城市独立完成 Manifest 收货和差异关闭 | Manifest 查询、扫描、差异、Case、关闭、custody | 三站并行正常/少货/多货/错站 |
| I05 调度与装车交接 | 各站发布任务并完成司机责任交接 | 波次草稿/发布/撤回、门禁、并发锁、LOAD 审批 | 无跨站/重复分配，责任一致 |
| I06 失败、回站与重派 | 未妥投件回到原站并形成下一动作 | 原因码、重试限制、RETURN、closeout、同站次日重派 | 领出=妥投+交回+批准持有 |
| I07 异常与回传 | Case 有 owner/SLA，回调死信可恢复 | Case 工作流、Outbox/attempt/ACK 查询、重放、审计 | 故障恢复、幂等和逐站队列通过 |
| I08 日终与试运营 | 各城市独立签字关站 | Reconciliation、钻取、结转、Runbook、培训 | 至少3城市连续5天，P0=0 |

## 5. `1.0 Stable Operations`：I09–I12（候选计划）

| 迭代 | 目标 | 主要交付 |
|---|---|---|
| I09 Partner Adapter | 接入第二个真实异构上游 | Adapter SPI、Partner 凭证/HMAC、字段/状态映射、契约沙箱、原文重放 |

## 6. `1.1 Operational Efficiency`：I13–I15（方向计划）

| 迭代 | 目标 | 主要交付 |
|---|---|---|
| I13 批量运营效率 | 减少重复人工操作 | 批量差异决策、批量波次、批量 Case、可撤销门禁和性能验证 |
| I14 配送顺序与通知 | 提升司机/收件体验 | 基础 stop sequence 建议、出车/到达/失败通知、模板和发送审计 |
| I15 SLA 与运营指标 | 用真实数据持续改善 | SLA 看板、首次妥投率、积压、回传延迟、导出和迭代基线对比 |

不在 `1.1` 自动承诺优化引擎、实时轨迹或总部汇总；只有指标证明价值后另开产品版本。

## 7. Definition of Done

每个迭代必须同步完成中英文需求/设计、Flyway（如有）、后端与运营 Web、权限/幂等/并发、单元/集成/API/Playwright/真实 MySQL E2E、演示证据、发布回滚说明和总结。未通过真实业务验收的功能不能仅因编译或测试通过而标记 Done。

`0.5` 的 P0 阻断项包括：路由错误、跨站点混用、责任链断裂、重复任务、POD 丢失、回调不可恢复和日终无法解释。路线优化、实时轨迹和高级报表不阻断 MOV。

## 8. R02.1 运营控制塔（新增必做 Gate）

R02 地图规划之后、R03 到仓实物之前插入 R02.1。它交付流程导航、今日控制塔、可行动指标/异常、下一步队列和地图可观察性。能力存在但运营无法发现、理解或排序，同样会阻塞试运营。R02.1 按 CT01–CT04 执行，完成标准见 [运营控制塔迭代计划](../iterations/iteration-r02-1-control-tower.md)。
